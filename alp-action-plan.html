Agent Load Protocol

ALP — Action Plan

Brainstorm → architecture → deliverables.
How ALP connects to SDK, MCP, API keys, GitHub, and every chat window.

1. The core insight

ALP is not an agent and not a tool runner. It is the universal manifest that tells any runtime — IDE, chat window, deployed UI — what an agent is, where it lives, and how to authenticate and call it.

SDK and MCP are the transport mechanisms that ALP sits on top of. The relationship is a stack, not a replacement:

Figure 1 — ALP as the format layer above MCP, SDK, and HTTP transports

  • MCP exposes tools only — callable functions the LLM can invoke. No concept of persona or memory.
  • SDK exposes the full agent — identity, memory, persona, tools — but is platform-specific.
  • ALP describes the whole agent (like SDK), and instructs the runtime where to find the MCP-compatible tool endpoints. ALP uses MCP as its tool-call transport.
Key mental model: MCP is to tools what ALP is to entire agents. ALP does not replace MCP — it wraps it with identity, persona, memory, and portability.

2. How ALP connects the agent from platform to chat window

The flow from "agent built on a platform" to "agent live in a chat window" is a four-step chain:

Figure 2 — Four steps from platform to live agent in a chat window

1
Build the agent on any platform

Relevance AI, Hive, LangChain, or from scratch. The platform is where persona, tools, and memory are configured.

2
Write or export agent.alp.json

The Agent Card is the portable description. Commit it to the GitHub repo. It is always safe to publish — no secrets ever enter the card.

3
Deploy the ALP Server

Run alp_server.py locally or deploy it to Railway / Fly.io / Render. This produces a public HTTPS URL that exposes /agent, /tools, /health, and (new) /persona.

4
Register the server URL in the target runtime

In Claude: add the URL to claude_desktop_config.json as an MCP server. In ChatGPT: configure as a custom GPT plugin. In VS Code / Cursor: add as an MCP server in settings. The agent is now live in the chat window.

3. Where the API key lives — the definitive answer

⚠️ The API key never enters the Agent Card. The card is a public document — safe to commit to GitHub. Secrets live in one place only: environment variables on the ALP Server.

Figure 3 — Secrets are injected by the ALP Server; the card and runtime never see raw keys

The card references secrets by name using auth_ref: 

{
  "name": "web_search",
  "description": "Search the web for current information.",
  "endpoint": "/tools/web_search",
  "auth_ref": "serper_api_key"       ← a name, not a value
}

At runtime, the ALP Server reads os.environ["SERPER_API_KEY"] and injects it into the tool call. The key never appears in the card or in the chat window's context.

API key placement per runtime:

RuntimeWhere the key goes
ALP Server (all cases).env file or deployment env vars (Railway, Fly.io, etc.)
Claude DesktopALP Server env only — Claude never receives the key
ChatGPT PluginALP Server env only — same principle
GitHub Actions (CI)GitHub Secrets → injected as env vars into the server on deploy
VS Code / CursorALP Server env — the IDE calls the server's endpoints, not the LLM API directly

4. Localizing the right agent when the platform has many products

When a platform hosts many agents (Relevance AI, Hive HoneyComb), the client needs to know which agent to load. ALP solves this at three levels:

  • URL as locator — one ALP Server URL per agent, or one server that accepts ?agent_id=…. The URL is the unique address.
  • id field — the agent's identifier inside the server.
  • platform.agent_id + platform.project_id — traces back to the source platform for provenance.

Action: Add a GET /agents listing endpoint to the ALP Server spec as an optional extension. When one server hosts multiple agents, this endpoint returns an array of cards. Marketplaces (HoneyComb) can read this to display a full catalog without running any agent.

GET /agents
→ [
    { "id": "code-reviewer", "name": "Code Reviewer", ... },
    { "id": "sales-researcher", "name": "Sales Researcher", ... }
  ]

5. How the agent loads as a "tool" in the chat window

Figure 4 — How persona, tools, and memory surface together in the chat window

When an ALP agent loads into a chat window, three things happen in sequence:

  1. The persona (/persona endpoint) is fetched and injected as the LLM's system prompt.
  2. The tools (/tools) are registered as MCP-compatible callable functions — this is what the LLM sees as "tools it can use".
  3. The memory config tells the runtime whether to persist conversation state between sessions.

The user experiences talking to an agent. The LLM experiences a system prompt and a toolset. The underlying mechanism is the ALP Client stitching them together.

6. The GitHub repo as the portable agent

The GitHub repo is the agent. It contains the card and the server. To make it live in any chat window: 

your-repo/
├── agent.alp.json          ← the Agent Card (safe to commit publicly)
├── alp_server.py           ← the ALP Server reference implementation
├── requirements.txt
├── .env.example            ← documents required secrets (no actual values)
└── deploy/
    ├── railway.json        ← one-click Railway deploy
    ├── fly.toml            ← Fly.io config
    └── render.yaml         ← Render config
1
Clone the repo

git clone https://github.com/RodrigoMvs123/agent-load-protocol

2
Set your secrets

cp .env.example .env — fill in your API keys. Never commit this file.

3
Deploy the ALP Server

Push to Railway (or Fly.io). The public URL is your agent's address: https://your-agent.railway.app

4
Register in the chat window of choice

Claude: add "url": "https://your-agent.railway.app/tools" to claude_desktop_config.json. Done.

7. The three surfaces where the agent lives

🏗️
Surface 1 — Platform client

Relevance AI, Hive UI, LangChain. The platform reads the Agent Card natively. ALP is the export/import format. The platform block in the card records provenance.

🖥️
Surface 2 — Deployed UI

Developer builds their own frontend. Calls POST /tools/{name} directly. Full control over UI, branding, and UX. The Agent Card is the API contract.

💬
Surface 3 — Chat window

Claude, ChatGPT, Gemini, VS Code, Perplexity. The ALP Server URL is registered as an MCP server. Lowest-friction surface for end users.

9. Key config snippets — ready to use

Registering in Claude Desktop

{
  "mcpServers": {
    "my-agent": {
      "url": "https://your-agent.railway.app/tools"
    }
  }
}

Registering in VS Code / Cursor (MCP extension)

{
  "mcp.servers": [
    {
      "name": "my-agent",
      "url": "https://your-agent.railway.app/tools"
    }
  ]
}

Minimal Agent Card (v0.2.0)

{
  "alp_version": "0.2.0",
  "id": "my-agent",
  "name": "My Agent",
  "description": "A portable AI agent.",
  "persona": "You are a helpful assistant specialised in ...",
  "llm": {
    "provider": "any",
    "model": "user-defined"
  },
  "tools": [
    {
      "name": "web_search",
      "description": "Search the web.",
      "endpoint": "/tools/web_search",
      "auth_ref": "serper_api_key"
    }
  ],
  "memory": { "enabled": false },
  "capabilities": ["tool-use", "web-search"],
  "server": {
    "url": "https://your-agent.railway.app",
    "transport": "http"
  }
}

Running the ALP Server locally

git clone https://github.com/RodrigoMvs123/agent-load-protocol
cd agent-load-protocol/reference/server/python
cp .env.example .env           # fill in your API keys
pip install -r requirements.txt
AGENT_CARD_PATH=../../examples/hello-agent/agent.alp.json python alp_server.py

.env.example template

# LLM provider keys (the ALP Server injects these — never put them in the card)
OPENAI_API_KEY=
ANTHROPIC_API_KEY=
GOOGLE_API_KEY=

# Tool-specific keys (referenced via auth_ref in the card)
SERPER_API_KEY=
GITHUB_TOKEN=

# Server config
PORT=8000
AGENT_CARD_PATH=../../examples/hello-agent/agent.alp.json

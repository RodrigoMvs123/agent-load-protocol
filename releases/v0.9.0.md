# ALP v0.9.0 — GitHub-native agent

**Released:** 2025 · Backward-compatible with v0.4.0. No breaking schema changes.

## What's new

### GitHub-native agent pattern

ALP agents can now live entirely in a GitHub repository and load
automatically into Kiro — with secrets managed by GitHub, deployment
triggered by GitHub Actions, and authentication handled by GitHub OAuth.

No manual server setup. No exposed API keys. One prompt to load.

### How it works

```
User in Kiro: "load my agent from github.com/user/my-agent-repo"
    ↓
GitHub MCP: get_file_contents → reads agent.alp.json from repo
    ↓
Kiro reads card, sees runtime.deploy block
    ↓
User authenticates with GitHub (OAuth — one time)
    ↓
GitHub MCP: workflow_dispatch → triggers deploy.yml in agent repo
    ↓
GitHub Actions: checks out repo, injects GitHub Secrets, deploys to Render
    ↓
Workflow outputs live URL
    ↓
Kiro reads URL, connects to /mcp
    ↓
User chats with their agent
```

### New `runtime.deploy` block in the Agent Card

```json
"runtime": {
  "deploy": {
    "trigger": "github_actions",
    "workflow": ".github/workflows/deploy.yml",
    "credentials": [
      {
        "ref": "GEMINI_API_KEY",
        "description": "Google Gemini API key for the LLM (free quota available)",
        "source": "github_secrets"
      },
      {
        "ref": "RENDER_API_KEY",
        "description": "Render deploy API key",
        "source": "github_secrets"
      }
    ]
  }
}
```

This block is informational — it tells the runtime (Kiro) how to
deploy the agent and what credentials are required. The actual secret
values never appear in the card.

### Why Gemini + Render

- **Google Gemini API** has a free quota tier, making this path accessible
  to any developer without upfront cost (get a key at aistudio.google.com)
- **Render** has a free tier for web services and a simple deploy API
  that integrates cleanly with GitHub Actions

### Relation to github/github-mcp-server issue #2299

Issue #2299 proposes a `get_agent_card` tool for the GitHub MCP Server
that would return a parsed, schema-validated Agent Card in one call.

v0.9.0 works **without** that tool using `get_file_contents` (already
available in the GitHub MCP Server today). If `get_agent_card` is
approved and shipped, the loading step becomes cleaner — but the
pattern and all example files remain identical either way.

## Migration from v0.4.0

No breaking changes. The `runtime` block is optional. All existing
cards continue to work without modification.

To use the GitHub-native pattern: add a `runtime.deploy` block to your
card and follow `examples/github-native-agent/README.md`.
